code-review-back

Un harnais local pour prouver qu'un skill charge les bonnes references, produit une review evaluable et detecte les regressions.

Le but n'est pas de produire une plateforme d'evaluation. Le but est de garder un cadre minimal, mesurable et evolutif pour ajouter des regles de review sans casser les regles existantes ni charger des fichiers inutiles.

100% local hors Copilot CLI 9 cas Java Juge LLM + check deterministe Selectivite des references Gate negatif

Architecture du cadre

Skill

skills/code-review-back/SKILL.md contient le routage selectif vers les references de regles.

References

rules/*.rules.md decrit les regles actives ou candidates, indexees dans rules/index.md.

Evals

evals/cases/* contient les diffs et attendus. evals/runner execute, extrait et gate.

skills/code-review-back/SKILL.md
rules/
  index.md
  services.rules.md
  controller.rules.md
  security.rules.md
  rgpd-trace-dcp.rules.md
evals/
  cases/<case-id>/{input.diff,expected.yml}
  runner/{run-review,extract-trace,validate-review,run-judge,gate}.sh
  results/

Role de chaque fichier

Le repo se lit comme une chaine de responsabilites. Les regles sont la source de verite, le skill route vers ces regles, les evals prouvent que le routage et la review tiennent.

Chemin Role Quand le toucher
skills/code-review-back/SKILL.md Point d'entree du skill. Il decrit la mission, le format de sortie et le chargement selectif des references. Quand un nouveau domaine de regle doit etre route ou quand le format de review change.
rules/index.md Table d'index des regles : ID, titre, fichier, statut. A chaque ajout, renommage, activation ou depreciation de regle.
rules/*.rules.md Reference detaillee d'une regle : detection, exclusions, risque, correctif, exemples, evals. Quand la norme projet evolue. C'est la source de verite normative.
decisions/ledger.md Journal des decisions : regles acceptees, rejetees, remappage d'anciens IDs. Quand une regle change de statut ou quand une decision doit rester tracable.
evals/cases/<case-id>/input.diff Entree testee par le skill. Peut etre un petit diff ou une classe Java complete. A chaque nouveau scenario de regression ou de validation d'une regle.
evals/cases/<case-id>/expected.yml Oracle du cas : findings attendus, interdits, maximum, fichiers de references attendus ou interdits. Quand l'objectif du cas ou la selectivite attendue change.
evals/runner/run-review.sh Lance Copilot CLI en non-interactif, active le skill et capture review.txt et trace.raw. Seulement si la surface CLI change.
evals/runner/extract-trace.sh Extrait tokens, fichiers lus et activation du skill depuis la trace. Quand le format de trace evolue ou qu'une nouvelle metrique devient observable.
evals/runner/validate-review.sh Controle deterministe les invariants mecaniques de la review. Quand le format de sortie attendu change.
evals/runner/run-judge.sh Lance le juge LLM et exige un JSON parsable. Quand le prompt ou le schema de juge change.
evals/runner/gate.sh Agrege justesse, selectivite, tokens et cas should_fail. Quand une nouvelle politique de blocage est decidee.
evals/results/FINDINGS.md Compte rendu factuel de calibration : ce que le dernier run reel a revele. Apres une campagne significative ou un changement de comportement observe.

Workflow d'execution

La chaine teste la plomberie complete, pas seulement la qualite de la reponse finale.

1. setup Verifie le CLI, capture l'aide, enregistre le repertoire de skills.
2. review Lance copilot -p, active le skill, capture review et trace JSONL.
3. trace Extrait tokens, fichiers lus et activation du skill depuis trace.raw.
4. juge Juge la validite semantique des findings avec un LLM qui doit rendre du JSON.
5. gate Agrege juge, check deterministe, selectivite et controle negatif. 
make eval
make case CASE=C8-medium-security-admin
make gate
make context

Deroule concret d'une analyse

Exemple avec C8-medium-security-admin, un controller admin complet qui doit declencher SEC-001 sans charger les references controller, service ou DCP.

1. Lancer le cas

make case CASE=C8-medium-security-admin

Le runner cree evals/results/C8-medium-security-admin/ et y depose tous les artefacts utiles pour comprendre le resultat.

2. Lire la synthese gate

case | justesse | selectivite | tokens
C8-medium-security-admin | PASS | PASS | in=null out=1287

Si une colonne echoue, on descend dans les artefacts du cas au lieu d'interpreter la sortie globale.

Artefact Question a se poser Signal attendu
review.txt Qu'a repondu le modele ? Les findings sont-ils lisibles et au bon format ? Lignes - [fichier:Lligne] ... (SEC-001), ou exactement AUCUN FINDING.
verdict.json Les findings sont-ils semantiquement vrais, ou juste bien formates ? "result": "PASS", avec matched, missed et reasons.
review-check.json Les invariants mecaniques passent-ils sans interpretation LLM ? Rule ID attendu present, fichier et fragment attendus, nombre de findings sous max_findings.
metrics.json Le skill a-t-il ete active ? Quels fichiers ont ete lus ? Quels tokens sont observes ? skill_activated vaut code-review-back, files_read contient la bonne regle.
trace.raw Si une mesure semble bizarre, quelle est la preuve brute ? Evenements JSONL skill, view, assistant.message.
expected.yml Le cas teste-t-il une chose claire ? Le referentiel attendu est-il le bon ? expected_findings, max_findings, context_expectations.
Regle de lecture : on commence par la table du gate, puis on ouvre l'artefact lie a la colonne en echec. Justesse en echec : review.txt, verdict.json, review-check.json. Selectivite en echec : metrics.json puis trace.raw.

Contrat d'un cas d'evaluation

Un cas reste petit cote schema. Le diff peut etre court ou contenir une classe complete, mais l'objectif d'evaluation doit rester explicite.

Exemple

expected_findings: []
max_findings: 0
context_expectations:
  exact_files_read:
    - rules/services.rules.md

Lecture

Trois axes, un champ par axe. Chaque champ peut faire echouer un cas que les autres laisseraient passer.

  • Justesse : expected_findings decrit ce qui doit etre trouve.
  • Bruit : max_findings plafonne le nombre de findings.
  • Selectivite : exact_files_read prouve que seul le bon referentiel a ete lu.
  • allowed_files_read remplace exact_files_read pour les vrais negatifs a contexte optionnel.
  • should_fail: true marque le controle negatif du harnais.

Qui juge quoi

Deux oracles, sans recouvrement volontaire. Chacun voit ce que l'autre ne peut pas voir.

Deterministe — le mecanique

review-check.json. Le rule_id, le fichier et les fragments attendus sont-ils presents ? Le nombre de findings tient-il sous max_findings ?

C'est un matcher de sous-chaines : gratuit, reproductible, et il ne se trompe jamais sur ce qu'il sait faire. Mais il est aveugle au sens : il verifie que les elements sont presents, jamais qu'ils ont un rapport entre eux.

Juge LLM — le semantique

verdict.json. Le finding affirme-t-il une violation ? La justification correspond-elle a la Detection de la regle, ou decrit-elle une Exclusion ? Le rule_id cite est-il le bon ?

Le juge ne recompte pas les findings : ce serait payer un appel modele pour un resultat deja acquis. Il ne fait que ce que lui seul sait faire.

Pourquoi les deux : cette review passe le deterministe sans broncher. 
- [SRV-001] src/main/java/demo/UserManager.java : la classe Service
  est parfaitement nommee, aucune violation detectee, rien a signaler
Le rule_id est la, le fichier est la, le fragment Service est la, il y a un seul finding. Elle conclut pourtant l'inverse exact de ce qui est attendu. Seul le juge peut la rejeter. C'est tout le cas C10-faux-pass : une trace figee qui rejoue cette review, et qui vire au rouge si le juge la laisse passer.

Ce qui est mesure

Activation

Le skill est prouve via session.skills_loaded, tool.execution_start et skillName.

Contexte

Les fichiers lus sont extraits des appels view. Le gate compare les fichiers de regles.

Tokens

outputTokens est extrait. Les input tokens restent null si le CLI ne les expose pas.

Le cadre ne simule pas une mesure manquante. Une donnee introuvable reste null ou fait echouer la selectivite si elle est necessaire au gate.

Ajouter une regle sans casser le cadre

Processus minimal

  • Ajouter ou modifier rules/<domaine>.rules.md.
  • Mettre a jour rules/index.md.
  • Declarer l'eval dans la ligne Evals:.
  • Ajouter un cas positif et si possible un vrai negatif proche.
  • Lancer make eval.

Regle de conception

Chaque cas doit avoir un objectif principal. Les diffs peuvent etre realistes, mais l'attendu doit rester lisible : quelle regle, quels faux positifs interdits, quels fichiers de reference autorises.

Les diffs medium ont revele une limite utile : un controller avec POST/PUT sur des ressources metier n'est pas un bon vrai negatif securite. Le fixture clean utilise donc un controller de catalogue public en lecture seule.

Limites assumees

  • Les input tokens ne sont pas exposes par la surface CLI observee.
  • Le juge LLM n'est pas l'unique oracle : le mecanique est verifie deterministiquement.
  • Le juge reste non-deterministe : il peut produire des faux FAIL. C'est le prix du seul controle semantique disponible.
  • La selectivite depend de la presence des appels view dans la trace.
  • Les classes completes peuvent produire plusieurs findings sur la meme regle, ce qui doit etre explicite via max_findings.
